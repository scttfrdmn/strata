# Multi-Version Layer Coexistence and LMod Integration

Design notes on running multiple versions of the same software simultaneously within a
Strata environment, and using LMod to switch between them at runtime.

---

## The Problem with the Current Design

Today, a layer's `build.sh` installs software directly into `$STRATA_PREFIX`:

```bash
"../gcc-${VERSION}/configure" \
  --prefix="${STRATA_PREFIX}" \   # ← root of the squashfs
  ...
```

`$STRATA_PREFIX` is the temp output directory from which the squashfs is created. Files
installed there appear at the **root** of the squashfs. When the layer is mounted at
`/strata/env` via OverlayFS, the result is:

```
/strata/env/bin/gcc        ← gcc 13.2.0
/strata/env/lib/gcc/
/strata/env/include/
```

If you add a gcc@14.2.0 layer using the same layout, the OverlayFS merge produces a
conflict — the layer with higher mount priority silently shadows the other. You get one
gcc, not two. The resolver's conflict detection (both layers provide `gcc`) would catch
this and refuse to resolve the lockfile.

To support multi-version coexistence, a structural change is needed.

---

## The Fix: Version-Prefixed Install Paths

Each layer should install into a **version-namespaced subdirectory** of `$STRATA_PREFIX`
rather than directly into its root. The pipeline provides a second environment variable:

```
STRATA_INSTALL_PREFIX = $STRATA_PREFIX/<name>/<version>
```

The build script uses `STRATA_INSTALL_PREFIX` for its `--prefix`:

```bash
# gcc/13.2.0/build.sh
"../gcc-${VERSION}/configure" \
  --prefix="${STRATA_INSTALL_PREFIX}" \   # ← /strata/out/gcc/13.2.0
  ...
```

```bash
# gcc/14.2.0/build.sh  (identical structure)
"../gcc-${VERSION}/configure" \
  --prefix="${STRATA_INSTALL_PREFIX}" \   # ← /strata/out/gcc/14.2.0
  ...
```

When the squashfs is created from `$STRATA_PREFIX`, it contains:

```
gcc/13.2.0/bin/gcc
gcc/13.2.0/bin/g++
gcc/13.2.0/lib/gcc/
gcc/13.2.0/include/
```

When mounted at `/strata/env`, these become:

```
/strata/env/gcc/13.2.0/bin/gcc
/strata/env/gcc/13.2.0/lib/
/strata/env/gcc/14.2.0/bin/gcc    ← no conflict
/strata/env/gcc/14.2.0/lib/
```

The two layers coexist in the OverlayFS without conflict. Neither is on `PATH` by default —
LMod handles activation.

---

## LMod Integration

Each layer ships its own LMod modulefile inside the squashfs at:

```
modulefiles/<name>/<version>.lua
```

Which mounts at:

```
/strata/env/modulefiles/gcc/13.2.0.lua
/strata/env/modulefiles/gcc/14.2.0.lua
```

### Example modulefile (auto-generated)

```lua
-- /strata/env/modulefiles/gcc/13.2.0.lua
-- Auto-generated by strata build from content manifest.
local version = "13.2.0"
local base    = "/strata/env/gcc/" .. version

whatis("GCC " .. version .. " — C, C++, Fortran compilers")
conflict("gcc")   -- only one gcc active at a time via module load/swap

prepend_path("PATH",              base .. "/bin")
prepend_path("LD_LIBRARY_PATH",   base .. "/lib64")
prepend_path("LD_LIBRARY_PATH",   base .. "/lib")
prepend_path("MANPATH",           base .. "/share/man")
prepend_path("INFOPATH",          base .. "/share/info")
prepend_path("PKG_CONFIG_PATH",   base .. "/lib/pkgconfig")
setenv("GCC_HOME",                base)
setenv("CC",                      base .. "/bin/gcc")
setenv("CXX",                     base .. "/bin/g++")
setenv("FC",                      base .. "/bin/gfortran")
```

### Profile that loads both versions

```yaml
name: compiler-dev
base:
  os: al2023

software:
  - gcc@13.2.0
  - gcc@14.2.0
  - python@3.12
```

The resolver allows both because:
1. They install to non-conflicting paths (`gcc/13.2.0/` and `gcc/14.2.0/`)
2. Neither is on `PATH` by default — no file-level conflict in the OverlayFS
3. The `conflict("gcc")` in the LMod modulefile prevents both being *active*
   simultaneously, which is the right behavior

### User workflow

```bash
# Environment boots — /strata/env is mounted, no gcc on PATH yet

module use /strata/env/modulefiles
module avail gcc

# Output:
# gcc/13.2.0
# gcc/14.2.0

module load gcc/13.2.0
gcc --version    # gcc (GCC) 13.2.0

module swap gcc/13.2.0 gcc/14.2.0
gcc --version    # gcc (GCC) 14.2.0
```

### Auto-enabling LMod

The strata-agent can write `/etc/profile.d/strata-modules.sh`:

```bash
# /etc/profile.d/strata-modules.sh
# Auto-generated by strata-agent
if command -v module &>/dev/null; then
    module use /strata/env/modulefiles
fi
```

If LMod is not present, this is a no-op. If LMod is present (either as a Strata layer or
pre-installed on the AMI), module files are available immediately on login.

## Implementation Requirements

### 1. Pipeline change (`internal/build/pipeline.go`)

Add `STRATA_INSTALL_PREFIX` to the env vars passed to the executor:

```go
installPrefix := filepath.Join(outputDir, recipe.Meta.Name, recipe.Meta.Version)
if err := os.MkdirAll(installPrefix, 0755); err != nil {
    return nil, fmt.Errorf("creating install prefix: %w", err)
}
env := []string{
    "STRATA_PREFIX="         + outputDir,
    "STRATA_INSTALL_PREFIX=" + installPrefix,   // ← new
    "STRATA_NCPUS="          + strconv.Itoa(runtime.NumCPU()),
    "STRATA_ARCH="           + arch,
    "STRATA_OUT="            + outputDir,
}
```

### 2. Build script convention

Build scripts should prefer `$STRATA_INSTALL_PREFIX` for their install prefix:

```bash
# Preferred (supports multi-version coexistence)
--prefix="${STRATA_INSTALL_PREFIX}"

# Legacy (single-version only, root of squashfs)
--prefix="${STRATA_PREFIX}"
```

For software that has no conflict risk (e.g., a one-off tool), `$STRATA_PREFIX` is still
valid. The convention is: use `$STRATA_INSTALL_PREFIX` for anything that other Strata
layers might also provide.

### 3. Modulefile generation (`strata build`)

After `make install`, the build pipeline generates a LMod modulefile from the content
manifest and writes it to `$STRATA_PREFIX/modulefiles/<name>/<version>.lua`. This is
included in the squashfs automatically.

The generator walks `$STRATA_INSTALL_PREFIX` and infers standard paths:
- `bin/` → `prepend_path("PATH", ...)`
- `lib/` or `lib64/` → `prepend_path("LD_LIBRARY_PATH", ...)`
- `lib/pkgconfig/` → `prepend_path("PKG_CONFIG_PATH", ...)`
- `share/man/` → `prepend_path("MANPATH", ...)`
- `share/info/` → `prepend_path("INFOPATH", ...)`
- `include/` → `prepend_path("CPATH", ...)`

Special variables (`GCC_HOME`, `PYTHON`, `R_HOME`, etc.) are declared in `meta.yaml`:

```yaml
# meta.yaml
modulefile_env:
  - var: GCC_HOME
    path: ""             # = $STRATA_INSTALL_PREFIX itself
  - var: CC
    path: "bin/gcc"
  - var: CXX
    path: "bin/g++"
  - var: FC
    path: "bin/gfortran"
```

### 4. Resolver change

The resolver's conflict detection currently rejects two layers that provide the same
capability name. With version-prefixed paths, a new resolution rule is needed:

> Two layers providing the same capability are allowed if neither installs files that
> overlap in the OverlayFS (i.e., both use `STRATA_INSTALL_PREFIX`-style layout).

This can be declared in `meta.yaml`:

```yaml
install_layout: versioned   # installs to name/version/ — coexistence allowed
# vs.
install_layout: flat        # installs to squashfs root — conflict on same provides
```

The default for new recipes should be `versioned`. Legacy/flat layers (e.g., cuda, which
installs a single fixed version) remain `flat`.

## Profiles: Default vs Explicit Activation

When a profile includes a versioned layer, it is available in the environment but not
active by default. The profile can declare a default:

```yaml
software:
  - gcc@13.2.0
  - gcc@14.2.0
  - python@3.12

defaults:            # loaded automatically at login
  - gcc@13.2.0
  - python@3.12
```

The strata-agent writes the defaults to `/etc/profile.d/strata-defaults.sh`:

```bash
module load gcc/13.2.0
module load python/3.12.0
```

Users can override: `module swap gcc/13.2.0 gcc/14.2.0` — and the environment persists
for the session.

## Relationship to Toolchain Formations

Multi-version coexistence is particularly valuable for compiler toolchains. A
`compiler-dev` formation might ship gcc@13 and gcc@14 together precisely so a developer
can test that their code compiles cleanly under both before publishing a Strata layer that
declares `build_requires: gcc@>=13`.

For production use, toolchain formations should pin a single version:

```yaml
# formations/foss-2024a.yaml
name: foss-2024a
type: toolchain
software:
  - gcc@13.2.0      # pinned — ABI stability matters for Tier 1 builds
  - openmpi@5.0
  - openblas@0.3.26
  - fftw@3.3.10
```

The LMod modulefile for `foss-2024a` would load all four in one step:

```lua
-- modulefiles/foss-2024a.lua
depends_on("gcc/13.2.0")
depends_on("openmpi/5.0")
depends_on("openblas/0.3.26")
depends_on("fftw/3.3.10")
whatis("foss-2024a toolchain: GCC 13.2 + OpenMPI 5.0 + OpenBLAS + FFTW")
```

Formation modulefiles are generated from the formation YAML — no manual authoring needed.

## Migration Path for Existing Recipes

The current gcc recipes use `--prefix="${STRATA_PREFIX}"` (flat layout). The built
squashfs for gcc@13.2.0 has files at the root. These layers work fine when there is only
one gcc in the environment. To support multi-version, the recipes need to be rebuilt with
`STRATA_INSTALL_PREFIX`. This is a **breaking change** to the layer artifact — the SHA256
will change. Rebuilt layers should be published as new registry entries; old layers remain
valid for profiles that don't need multi-version.

The v0.10.0 build pipeline (EC2-orchestrated, `STRATA_INSTALL_PREFIX` support) is the
natural point to rebuild Tier 0 layers with the new layout.
